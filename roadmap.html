Roadmap

Where RPCortex is headed. Updated as versions ship and plans solidify.

✓
v0.8.1 — Nebula Shipped

The final stable release of the β8 "Nebula" line. Closed out the beta backlog: unlimited WiFi profiles, manual recovery mode, --help flags, pkg commands, tilde expansion everywhere, password masking, a functional Program_Execution toggle, echo file redirection, and POST .mpy fallback.

🔵
v0.9.0 — Pulsar Stable · Current

The first release of the β9 "Pulsar" series and the direct successor to v0.8.1. A stability-and-quality-of-life release: persistent aliases, over-the-air updates, safer large-file operations, and a leaner, faster shell. The benchmark and system-info tools became real packages, and a portable compiled .mpy image now ships alongside source.

Shell & Filesystem
  • Persistent aliases — survive reboots via aliases.cfg
  • Streamed cp / mv — 1 KB chunks, no more large-file OOM; mv uses an instant same-FS rename
  • Relative paths for cp / mv / rename
  • Multi-file cat, du (tree size), watch (periodic re-run)
  • date set — set the RTC so log timestamps are correct
  • Configurable prompt hostname — reg set System.Device_ID <name>
Updates & Packages
  • Over-the-air updates — update check / update online install the latest release over WiFi
  • fetch and bench are packages now (PicoFetch / NebulaMark) — updatable via pkg upgrade
  • pkg upgrade handles built-in packages and preserves their registry keys
  • Version & codename never drift — synced from the running code on every boot
Performance & Build
  • No typing lag — per-keystroke completion is zero-I/O; filesystem only touched on Tab
  • No output lag — log writes batched (flush every 8 lines / on warnings)
  • Portable compiled image — architecture-neutral .mpy build (~44% smaller, faster import)
Pipes (|) and && / || were intentionally deferred — they need a per-command exit-code convention to be done correctly. They lead the v0.9.1 plan below.
🟡
v0.9.1 Next

The next pass focuses on shell power and automation — the things that turn RPCortex from an interactive shell into something you can script and leave running.

  • Shell pipes — cat f | grep ERR; the unblocking item for everything else
  • Startup tasks — run commands / scripts automatically after login
  • Scheduled tasks — repeat at intervals via software uptime timer
  • First-cut scripting — variables, conditionals, loops in .rps scripts
  • Expanded recovery tools — fsck-style checks, registry repair, log export
  • Download progress bars + extended curl flags
Shell & Scripting
  • Shell pipes — cat file | grep ERROR, built on a real per-command exit-code / stdout-capture convention
  • && / || chaining — conditional sequencing, the companion to pipes
  • Custom scripting language — first cut: variables, conditionals, loops, and ~ expansion, tightly integrated with Launchpad
Task Scheduling & Background
  • Startup tasks — run commands, scripts, or packages automatically after login
  • Scheduled tasks — run things at intervals using a software uptime offset (no battery-backed RTC needed)
  • Unattended / background mode — long-running tasks (tail a log, watch GPIO) without holding the shell
Recovery & Hardware
  • Expanded recovery tools — diagnostics and repair commands inside recovery mode (fsck-style checks, registry repair, log export)
  • GPIO control via registry — toggle pins to reset a hung peripheral without a reboot
Networking & UX
  • Download progress bars — [####----] 47% for wget/curl/pkg install via Content-Length
  • Extended curl flags — -X, -d, -H, -o, -s, -I, timeout
  • OS personalisation — owner name, device name, timezone
Memory & Performance
  • Lazy / ephemeral imports — evict non-critical modules after use, then gc.collect()
  • Module eviction under memory pressure with automatic retry
  • Avoid string concatenation in hot paths to cut fragmentation
  • Targeted Pico W RAM tuning (~200 KB free ceiling)
The exact split between v0.9.1 and later v0.9.x point releases will firm up as work lands. This is the current plan, not a firm commitment.
⚪
v0.9.x → v1.0 Future

The road to v1.0. The headline milestone is real concurrency; everything here is tentative and depends on the multitasking foundation landing first.

Concurrency
  • First uasyncio integration — cooperative multitasking for background tasks alongside the shell
  • Proper multi-threading & background services — the core architectural goal before v1.0; heavily tested
  • Multiple terminals & tabbing — PowerShell-style session management, once multitasking is stable
  • SSH-style shell over TCP — a lightweight listener that hands out a Launchpad session; needs uasyncio
Apps & Developer Tools
  • TUI framework — reusable adaptive box-draw components for building terminal apps
  • System apps on the TUI framework — rebuild settings, edit, and friends
  • Dev packages — git (clone / fetch / push), a speedtest package, and more
  • SD card support — init package, surfaced as a startup task
Build & Portability
  • Frozen modules / custom .uf2 firmware build (feasibility)
  • Windows MicroPython port stubs for off-device debugging (conditional machine imports, relative paths)
  • Mouse support in the TUI (terminal-dependent)
  • Viper / native optimisation — deferred; would sacrifice non-RP2 portability
Package ideas Anytime

Packages install and update independently of the OS, so these don’t gate any release — they can land whenever someone builds them. The OS already has the networking and hardware primitives most of them need. Roughly ordered by usefulness × ease.

  • ntp — sync the clock from an NTP server over WiFi (no more clock reset on power loss). Best first package.
  • dht — read temperature & humidity from a DHT11/DHT22; the most common Pico sensor.
  • i2cscan — scan the I2C bus and name detected devices; the first thing you do with a new sensor.
  • gpio — set / read / PWM pins straight from the shell.
  • calc — evaluate math from the prompt; base conversions (hex, bin).
  • backup — archive & restore your home directory and registry before a reset or migration.
  • httpd — serve a live status page (CPU / RAM / uptime / temp) over WiFi.
  • mqtt — publish / subscribe so RPCortex can talk to Home Assistant, Node-RED, and friends.
  • speedtest — measure WiFi up/down throughput in Mbps.
  • git — clone / fetch repositories (read-mostly); the heaviest of the set.
Got an idea for a package? Open an issue or drop it in Discord — see the Package Dev guide to build your own.
